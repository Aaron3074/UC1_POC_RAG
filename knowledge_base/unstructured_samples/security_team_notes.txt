AUTH-SVC / KEYCLOAK SECURITY NOTES
- keycloak version: 22.0.5 (current), upgrade to 23.x scheduled Q3 2025
- AUTH_MAX_LOGIN_ATTEMPTS = 5 (account locks after 5 failed logins)
  this is the most common cause of "user can't log in" tickets from the help desk
  to unlock: keycloak admin console → Users → find user → Sessions → Clear Sessions + unlock
  URL: https://10.10.1.50:8443/auth/admin (requires admin credentials, request via IT helpdesk)
- auth logs are fed to QRadar via syslog. QRadar rules alert on:
  >50 failed logins in 5 minutes from single IP (brute force detection)
  >1000 failed logins across all users in 15 minutes (credential stuffing detection)
  alert destination: security-ops@company.com + PagerDuty security rotation

KEYCLOAK REALMS
  production realm: company-prod
  test realm: company-test (not accessible from prod services)
  all service accounts use client credentials flow (not password flow)
  service account client IDs follow naming: svc-{servicename}-prod (e.g. svc-payments-gw-prod)

ACCESS CONTROL (IAM)
- all human access to prod follows break-glass procedure via CyberArk PAM
  no direct SSH with static credentials to prod. all sessions proxied through bastion: 10.10.50.5
  bastion host requires MFA (TOTP via Keycloak)
  session recording enabled - all prod SSH sessions are recorded and retained for 90 days
- service-to-service auth: OAuth2 client credentials via AUTH-SVC
  payments GW → CBS: uses svc-payments-gw-prod client
  fraud engine → payments GW: uses svc-fraud-engine-prod client
  all service accounts rotate client secrets every 90 days (automated via vault)
