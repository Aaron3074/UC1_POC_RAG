SHEET 2 — ROLLBACK DETAIL
===========================

ROLLBACK: REL-2025-009 — FRAUD-ENGINE v3.3.0
  Date: 2025-03-13 22:00 deploy → 2025-03-14 00:47 rollback (2hr 47min)
  Reason: v3.3.0 introduced a new feature flag mechanism that caused model scoring
           to return NaN for transactions with NULL merchant_category_code.
           PAYMENTS-GW_FRAUD_THRESHOLD comparison with NaN evaluates to False in Python —
           meaning all transactions with NULL merchant codes bypassed fraud checks.
  Detection: Risk Tech monitoring dashboard showed fraud score null rate spike to 12% at 23:15.
             Risk analyst escalated at 23:41. Rollback decision at 00:30.
  Impact: ~1,800 transactions processed without fraud scoring (merchant_category_code NULL).
          All 1,800 transactions flagged for manual review. 3 confirmed fraudulent.
          Financial exposure: $4,200 (recovered via chargeback).
  Fix: v3.3.1 deployed 2025-03-20 with NULL-safe comparison and unit test coverage for NULL inputs.
  Process improvement: Fraud scoring null rate added as automated deployment gate.
                       If null rate >1% in first 15 min post-deploy, auto-rollback triggers.

SHEET 3 — RELEASE NOTES SUMMARY
===========================

REL-2025-002 — PAYMENTS-GW v2.8.0 (Feature Release)
  New features:
    - PAYMENTS-GW_MAX_PAYLOAD_SIZE_MB configurable at runtime (previously required restart)
    - Payment retry logic refactored: exponential backoff replaces fixed 30s retry
    - New /payments/metrics Prometheus endpoint for Datadog scraping
  Bug fixes:
    - Fixed race condition where concurrent retries could duplicate transactions (rare edge case)
    - SWIFT_TIMEOUT_SEC now correctly applied per-transaction (was global previously)
  Config changes: None required. Backward compatible.
  Rollback procedure: systemctl stop payments-gw; deploy v2.7.3 package; systemctl start payments-gw

REL-2025-006 — CBS v4.2.0 (Major Release)
  New features:
    - Beneficiary account type field added to transaction schema (column: beneficiary_account_type)
      NOTE: Data Engineering notified per new schema change process (lessons from INC-2024-0892)
    - Bulk payment processing API endpoint (up to 500 transactions per request)
    - Enhanced audit logging for high-value transactions (>$100,000)
  Performance improvements:
    - CBS_DB_POOL_SIZE default changed from 50 to 100 (permanent fix from INC-2024-1201)
    - Oracle query optimisation for account balance lookups (23% latency improvement)
  Config changes:
    - CBS_DB_POOL_SIZE: 50 → 100 (DBA approved, already pre-deployed as hotfix REL-2025-001)
  Data Engineering sign-off: CONFIRMED (schema change notification sent 2025-01-28)

REL-2025-007 — Kafka v3.5.1 (Infrastructure Upgrade)
  Purpose: Kafka version upgrade from 3.4.0 to 3.5.1
  Key changes:
    - Performance improvements in log segment management (addresses INC-2025-0044)
    - Improved consumer group rebalance algorithm (reduces rebalance time by ~40%)
    - KRaft mode still not enabled (Zookeeper still in use — see ADR-008)
  Downtime: Zero downtime achieved via rolling upgrade (one broker at a time)
  Testing: Full consumer lag and throughput test in staging confirmed no regression

REL-2025-012 — SWIFT-ADAPTER v2.1.0 (Feature Release)
  New features:
    - SWIFT ISO 20022 message format support (MT → MX migration readiness)
    - Compliance filter updated to OFAC list as of Q1 2025
    - SWIFT_MAX_TRANSFER_LIMIT_USD now configurable per currency pair
  Note: ISO 20022 support is passive — MX format not yet live. Active from Q3 2025.

SHEET 5 — CAB PROCESS SUMMARY
===========================

Change Advisory Board (CAB) process (updated February 2025):

STANDARD CHANGE (low risk):
  Examples: Bug fixes, config changes with documented precedent
  Approval: Application owner + one peer reviewer
  Lead time: 2 business days
  CAB meeting: Not required

SIGNIFICANT CHANGE (medium risk):
  Examples: Feature releases, infrastructure changes, new integrations
  Approval: Application owner + Platform Engineering + CAB quorum (3 of 5 members)
  Lead time: 5 business days
  CAB meeting: Required

EMERGENCY CHANGE (high urgency):
  Examples: Security hotfix, active incident remediation
  Approval: CTO + Security lead (verbal OK accepted, documented post-hoc)
  Lead time: As needed
  CAB meeting: Post-change review within 24 hours

DECEMBER FREEZE EXCEPTION:
  Only security patches and active incident remediation permitted December 1–31.
  All other changes deferred to January. Exception requires CTO sign-off.
