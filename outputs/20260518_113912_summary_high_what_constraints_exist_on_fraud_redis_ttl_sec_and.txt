  [MODEL PARAMETER CONSTRAINTS]
    MODEL PARAMETER CONSTRAINTS
    FRAUD_MODEL_VERSION = "3.2.1" (in env, updated by deploy script)
    FRAUD_SCORE_THRESHOLD = 0.75 (scores above this trigger fraud hold)
    - DO NOT change this without risk committee approval
    - last changed: Q3 2023 after false positive rate review
    FRAUD_KAFKA_CONSUMER_GROUP = "fraud-scoring-cg-prod" (do not rename, kafka offset tracking depends on this)
    if you rename the consumer group, offset tracking is lost and the fraud engine will reprocess
    every message in the partition from the beginning. this caused INC-2024-0991 in september 2024.
    FRAUD_REDIS_TTL_SEC = 3600
    - redis cache TTL for fraud score results. set to 3600 by platform eng.
    - risk tech has requested reducing to 1800 for Q2 to improve freshness of repeat transaction scoring
    - platform eng said NO until redis auto-scaling is confirmed (sentinel config done, load test pending)
    - do NOT change this even if risk tech asks. they know the constraint.
    FRAUD_BATCH_SIZE = 500 (kafka consumer batch size)
    FRAUD_MAX_RETRIES = 3 (retry on scoring failure before dead-letter queue)

  [KNOWN ISSUES AND GOTCHAS]
    KNOWN ISSUES AND GOTCHAS
    - FRAUD_REDIS_TTL_SEC sensitivity: lower TTL = more cache misses = higher latency on payments GW
    at 900 TTL, payments GW latency spiked from 120ms to 380ms in load test (jan 2025)
    this is why platform eng wont reduce TTL until auto-scaling is proven
    - kafka consumer group rename caused INC-2024-0991 (offset reset, reprocessing backlog)
    rule: FRAUD_KAFKA_CONSUMER_GROUP is immutable in prod
    - model warm-up: fraud engine needs 3-5 minutes after deploy before scoring performance stabilises
    during warm-up, FRAUD_SCORE_THRESHOLD is temporarily raised to 0.85 to reduce false positives
    this is handled automatically by the deploy script. dont panic if you see fewer holds in first 5 min.
    - v3.4.0 specifically: the new model uses an updated feature vector that includes beneficiary_account_type
    this field comes from CBS via the payments gateway. CBS v4.2.0+ is required for this field to be present.
    CBS is already on v4.2.1 in prod so this dependency is met. just noting it.

    possible causes: fraud engine CPU maxed out (94% peak is normal for it)

    or FRAUD-RETRAIN-001 quarterly job is running (3 hours, spikes CPU)

    run kafka-log-dirs.sh to find large topics

  [kafka - stuff you need to know]
    no formal doc for this. just writing down what i know before i go on leave.
    CRITICAL THINGS:
    1. never change FRAUD_KAFKA_CONSUMER_GROUP in prod without risk tech sign-off
    changing consumer group = reset offset = reprocess millions of events = fraud engine dies
    2. replication factor is 3, min ISR is 2
    means cluster survives 1 broker going down. do NOT restart all 3 at once.
    3. kafka brokers: 10.10.1.70, 10.10.1.71, 10.10.1.72 all on port 9092
    SSL port is 9093 for cross-zone traffic
    4. monitoring is confluent control center: http://10.10.50.40:9021 (mgmt-net only)
    also zabbix has kafka broker health checks
    5. disk space is the main operational risk. INC-2024-0234 and INC-2025-0044 both disk.
    kafka 3.5.1 has better log segment handling. still watch it.

  [things that went wrong with kafka]
    INC-2024-0234 (march 2024, SEV-3):
    broker-2 disk space ran out. log segments accumulated.
    auto-recovered but took 31 min. added disk alert to zabbix after this.
    INC-2025-0044 (jan 2025, SEV-3):
    same thing. broker-2 disk again. 18 min.

    hasnt happened since. but still watch it.

    someone in dev environment accidentally ran consumer with --from-beginning flag.
    started consuming all historical messages from fraud.transaction.events.
    noticed in confluent control center. killed it before it caused lag.
    lesson: never run --from-beginning in prod. never change consumer group offsets manually.
