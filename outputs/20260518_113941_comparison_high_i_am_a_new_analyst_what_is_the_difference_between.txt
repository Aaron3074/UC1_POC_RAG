  ▸ Q: what's the difference between CBS and Payments Gateway?
      Q: what's the difference between CBS and Payments Gateway?
      A: CBS is the backend account ledger. Payments Gateway (PAYMENTS-GW) is the transaction processor.
    → When a payment happens: Payments GW receives it → calls Fraud Engine for a risk score →
      calls CBS to debit/credit the accounts → returns confirmation.
      CBS doesn't talk to the outside world. Payments GW does.
      PAYMENTS-GW host: 10.10.1.20, Port: 9000.

  ▸ Q: what is the SLA for CBS / payments?
      Q: what is the SLA for CBS / payments?
    → A: CBS SLA: 99.9% availability (target), 99.87% actual Q1 2025 (below target due to INC-2024-1201)
    → Payments GW SLA: 99.95% target, 99.91% actual Q1 2025
    → Full SLA register is in sla_register_and_actuals.txt in the knowledge base.

  ▸ Q: what is CBS?
      Q: what is CBS?
      A: Core Banking System. It's the mainframe-era system that holds all account data, balances, and
      transaction history. When people say "the core", "the mainframe", or "account system" they mean CBS.
      Host: 10.10.1.10, Port: 8080. API endpoint: http://10.10.1.10:8080/api/v2/
      CBS is owned by the CBS team (cbs-ops@company.com). Platform eng does the infra.

    → 2. Check user's role in Keycloak Admin Console → Users → [username] → Role Mappings
      3. Verify the correct role is assigned for the application they need:
      analyst role       → REPORTING-SVC read, DWH read (restricted)

      5. If urgent: Security team can provision same-day for production incidents
      ESCALATE: security-ops@company.com

  ▸ ISSUE: Payments Gateway not processing — HTTP 503/504
      STEPS:
    → 1. Check health: GET http://10.10.1.20:9000/payments/health (expected: status ok)
    → 2. SSH to 10.10.1.20 from MGMT-NET. Check: systemctl status payments-gw
      3. Check FRAUD-ENGINE: curl http://10.10.1.30:8500/fraud/score/health
      PAYMENTS-GW calls FRAUD-ENGINE synchronously. If Fraud Engine is down, payments are blocked.
      4. Check CBS: curl http://10.10.1.10:8080/api/v2/health
      5. Review logs: /var/log/payments-gw/app.log — look for ConnectionRefused or Timeout errors.
      ESCALATE: If unresolved 15 minutes, page PAYMENTS-ONCALL (PagerDuty).

  ▸ ISSUE: CBS returning 503 — core banking unavailable — mainframe down
      SYMPTOMS:
      - HTTP 503 from CBS API at http://10.10.1.10:8080/api/v2
      - Payments failing because CBS is unavailable
      - "Core banking unreachable" errors in PAYMENTS-GW logs
      STEPS:
      1. Check CBS health: curl http://10.10.1.10:8080/api/v2/health
    → 2. SSH to 10.10.1.10 from MGMT-NET: systemctl status cbs
    → 3. Check Oracle DB connection pool — CBS_DB_POOL_SIZE default is 100 (was 50 before INC-2024-1201).
      If pool exhausted: DBA team can increase temporarily. Alert at 80% pool utilisation should have fired.
      4. Check Oracle DB: sqlplus / as sysdba (from CBS host). Look for lock contention or tablespace full.
      5. Check CBS_BATCH_WINDOW_START — if during batch window (23:00-01:00), CBS may be under high load.
      Do not restart CBS during batch window without CBS Team approval.
      6. Review logs: /var/log/cbs/app.log
      ESCALATE: cbs-ops@company.com immediately. SEV-1 — CBS down affects all payment processing.

[5] Parameter Notes From Call
     File: parameter_notes_from_call.txt  (.TXT)  |  Folder: unstructured_samples

  ▸ PAYMENTS GATEWAY:
      PAYMENTS GATEWAY:
      - PAYMENTS-GW_TXN_TIMEOUT = 30000ms (30 seconds). multiply by CBS_MAX_TXN_RETRY and you get
      90 sec worst case for a transaction to fail completely.
      - PAYMENTS-GW_MAX_PAYLOAD_SIZE_MB = 5.0. requests above this get HTTP 413.

  ▸ CORE BANKING (CBS) stuff:
      CORE BANKING (CBS) stuff:
      - CBS_DB_POOL_SIZE was 50, is now 100 (fixed after the dec 2024 incident where we ran out of
      connections during peak load). if you want to go above 150 you need DBA approval.
      - CBS_SESSION_TIMEOUT_SEC = 1800 (30 min). PCI-DSS says minimum 900 seconds (15 min), max
      3600 (60 min). dont go below 900 or youll fail the audit.
      - CBS_BATCH_WINDOW_START = 23:00 - this is when EOD reconciliation kicks off. the DWH ETL
      starts at 23:30 so there's a 30 min buffer. dont change these without telling data eng.
    → - CBS_MAX_TXN_RETRY = 3. dont set above 5 or you risk duplicate transactions.

  ▸ FRAUD ENGINE stuff:
      FRAUD ENGINE stuff:
      - FRAUD_REDIS_TTL_SEC is currently 3600 (one hour). DO NOT change this without platform eng
    → sign off. when risk tech changed it to 300 in dec 2024 redis ran out of memory and we had
      a 138 min outage. the redis memory limit is 32GB and at 300sec TTL we were writing 12x more
      frequently than the memory could handle.
      - FRAUD_MODEL_VERSION is currently v3.2.1 (was briefly v3.3.0 but that got rolled back because
      of the NaN bug with null merchant codes). v3.4.0 is coming in may 2025.
    → - FRAUD_KAFKA_CONSUMER_GROUP = "fraud-engine-cg-prod" - if you change this it resets the
      kafka offset and reprocesses all historical messages. bad idea in prod. coordinate first.
    → - PAYMENTS-GW_FRAUD_THRESHOLD = 0.75 by default. if you lower it you get more false positives
    → (legitimate payments getting blocked). if risk tech wants to change this they need to tell
      platform eng and payments team first.

      contact for parameter questions: platform-eng@company.com or raise in #platform-engineering slack
