    Q: how do i know if the fraud model is performing well after an update?
    A: check FRAUD_MODEL_VERSION first to confirm which version is active.
    model performance metrics are in the fraud engine ML dashboard (risk tech team has access).
    key metric: null score rate. if null rate exceeds 1% post-deploy, auto-rollback fires.
    after the v3.3.0 incident (march 2025) where null merchant codes bypassed fraud checks,
    we added automated null rate gate in the deployment pipeline.
    if you see legitimate payments being blocked at unusual rates: PAYMENTS-GW_FRAUD_THRESHOLD
    is 0.75 by default. any change requires Risk Tech + Payments Team sign-off + CAB.

  [---]
    ---
    Q: can i query the kafka topics directly?
    A: depends on your role. if youre on MGMT-NET you can use kafka-console-consumer but be careful.
    do NOT change or commit offsets manually. do NOT reset the consumer group (FRAUD_KAFKA_CONSUMER_GROUP).
    resetting the consumer group on fraud.transaction.events would cause the fraud engine to
    reprocess 7 days worth of transactions - millions of events - and overwhelm the system.
    if you need to inspect a topic for debugging, read-only consumer is fine, just dont touch offsets.

  [---]
    ---
    Q: the payments portal shows "service temporarily unavailable". what does that mean?
    A: most likely one of: payments GW is down, fraud engine is down (payments GW calls it
    synchronously), or CBS is down. check in that order:
    curl http://10.10.1.20:9000/payments/health
    curl http://10.10.1.30:8500/fraud/score/health
    curl http://10.10.1.10:8080/api/v2/health
    if you cant reach any of them, youre not on MGMT-NET. need VPN.

  [---]

  [current consumer groups to never change without coordination:]
    current consumer groups to never change without coordination:
    - fraud-engine-cg-prod (fraud engine consuming fraud.transaction.events)
    - whatever groups reporting-svc uses for cbs.account.updates

  [just a reminder - do NOT change FRAUD_KAFKA_CONSUMER_GROUP or any other kafka co]
    just a reminder - do NOT change FRAUD_KAFKA_CONSUMER_GROUP or any other kafka consumer group
    ID in production without coordinating with the owning team first. changing the consumer group
    resets the kafka offset to the beginning of the topic. for fraud.transaction.events (12 partitions,
    7 day retention) this means reprocessing potentially millions of historical payment events.
    this would overwhelm the fraud engine. not good. coordinate first. always.

  [just to document what we learnt from the march 2025 incident (INC-2025-0203, v3.]
    just to document what we learnt from the march 2025 incident (INC-2025-0203, v3.3.0):
    - never merge a feature flag mechanism without comprehensive null testing
    - PAYMENTS-GW_FRAUD_THRESHOLD comparison with NaN evaluates to False in python
    - this means null merchant code = no fraud check = transaction goes through
    - 1800 transactions were unscored, 3 were actual fraud, $4200 exposure
    - we rolled back to v3.2.1 same night (REL-2025-009)
    - v3.3.1 fixed with null-safe comparison and added unit test for null inputs

  [1. any new column added to CBS transaction table must be notified to data eng BE]
    1. any new column added to CBS transaction table must be notified to data eng BEFORE deployment
    2. use the #schema-changes slack channel
    3. data eng gets minimum 3 days notice to update ETL mappings
    4. CBS release checklist now has a mandatory field: "data eng notified Y/N"
    5. if not notified and ETL breaks, CBS team owns the P1 response

  [MODEL PARAMETER CONSTRAINTS]
    MODEL PARAMETER CONSTRAINTS
    FRAUD_MODEL_VERSION = "3.2.1" (in env, updated by deploy script)
    FRAUD_SCORE_THRESHOLD = 0.75 (scores above this trigger fraud hold)
    - DO NOT change this without risk committee approval
    - last changed: Q3 2023 after false positive rate review
    FRAUD_KAFKA_CONSUMER_GROUP = "fraud-scoring-cg-prod" (do not rename, kafka offset tracking depends on this)
    if you rename the consumer group, offset tracking is lost and the fraud engine will reprocess
    every message in the partition from the beginning. this caused INC-2024-0991 in september 2024.
    FRAUD_REDIS_TTL_SEC = 3600
    - redis cache TTL for fraud score results. set to 3600 by platform eng.
    - risk tech has requested reducing to 1800 for Q2 to improve freshness of repeat transaction scoring
    - platform eng said NO until redis auto-scaling is confirmed (sentinel config done, load test pending)
    - do NOT change this even if risk tech asks. they know the constraint.
    FRAUD_BATCH_SIZE = 500 (kafka consumer batch size)
    FRAUD_MAX_RETRIES = 3 (retry on scoring failure before dead-letter queue)

  [fraud model deployment notes - internal - risk tech team]
    fraud model deployment notes - internal - risk tech team
    compiled by: risk tech lead
    covers: fraud-engine model versioning, deployment gates, known constraints
    last updated: april 2025

  [KNOWN ISSUES AND GOTCHAS]
    KNOWN ISSUES AND GOTCHAS
    - FRAUD_REDIS_TTL_SEC sensitivity: lower TTL = more cache misses = higher latency on payments GW
    at 900 TTL, payments GW latency spiked from 120ms to 380ms in load test (jan 2025)
    this is why platform eng wont reduce TTL until auto-scaling is proven
    - kafka consumer group rename caused INC-2024-0991 (offset reset, reprocessing backlog)
    rule: FRAUD_KAFKA_CONSUMER_GROUP is immutable in prod
    - model warm-up: fraud engine needs 3-5 minutes after deploy before scoring performance stabilises
    during warm-up, FRAUD_SCORE_THRESHOLD is temporarily raised to 0.85 to reduce false positives
    this is handled automatically by the deploy script. dont panic if you see fewer holds in first 5 min.
    - v3.4.0 specifically: the new model uses an updated feature vector that includes beneficiary_account_type
    this field comes from CBS via the payments gateway. CBS v4.2.0+ is required for this field to be present.
    CBS is already on v4.2.1 in prod so this dependency is met. just noting it.
