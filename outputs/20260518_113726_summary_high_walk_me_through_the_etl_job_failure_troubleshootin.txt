  [ISSUE: Reporting Service slow — timeout on report generation]
    STEPS:
    1. Check DWH query performance — review query plan in Snowflake console.
    2. Verify ETL-001 completed — stale DWH data causes report retries.
    3. Check resource usage: ssh 10.10.1.40, run top or htop.
    4. Check if CBS_BATCH_WINDOW_START overlaps with RPT-001 start time (I/O contention).
    ESCALATE: bi-ops@company.com

  [ISSUE: ETL-001 job failed — DWH showing stale data]
    STEPS:
    1. Check Airflow at http://10.10.50.30:8081 — find dag_cbs_to_dwh, review task logs.
    2. Common cause 1: EOD-001 did not complete. Check if /data/exports/eod/YYYYMMDD_eod.csv exists.
    3. Common cause 2: Snowflake DWH unreachable. Test connectivity from ETL host.
    4. Common cause 3: Schema mismatch — CBS added column not mapped. Check logs for KeyError.
    5. Manual re-trigger: In Airflow UI, clear and re-run dag_cbs_to_dwh for the failed date.
    ESCALATE: Data Engineering on-call if retry fails (PagerDuty: DE-ONCALL).

  [ISSUE: Kafka consumer lag high — Fraud Engine falling behind]
    SYMPTOMS:
    - Fraud scoring latency increasing
    - Consumer lag on fraud.transaction.events > 10,000 messages
    - PAYMENTS-GW waiting for fraud scores (transaction latency spike)
    STEPS:
    1. Check consumer lag: kafka-consumer-groups.sh --bootstrap-server 10.10.1.70:9092 --describe --group fraud-engine-cg-prod
    High lag (>10,000): Fraud Engine is falling behind.
    2. Check Fraud Engine CPU: ssh 10.10.1.30, run top. If CPU near 100% during retraining (FRAUD-RETRAIN-001), lag will recover after job completes.
    3. Check FRAUD_REDIS_TTL_SEC — if recently reduced, recompute load may be overwhelming the engine.
    4. Check Confluent Control Center: http://10.10.50.40:9021 (MGMT-NET only)
    5. Do NOT change FRAUD_KAFKA_CONSUMER_GROUP — this resets offset and causes reprocessing of millions of events.
    ESCALATE: risktech@company.com + platform-eng@company.com

  [ISSUE: CBS returning 503 — core banking unavailable — mainframe down]
    SYMPTOMS:
    - HTTP 503 from CBS API at http://10.10.1.10:8080/api/v2
    - Payments failing because CBS is unavailable
    - "Core banking unreachable" errors in PAYMENTS-GW logs
    STEPS:
    1. Check CBS health: curl http://10.10.1.10:8080/api/v2/health
    2. SSH to 10.10.1.10 from MGMT-NET: systemctl status cbs
    3. Check Oracle DB connection pool — CBS_DB_POOL_SIZE default is 100 (was 50 before INC-2024-1201).
    If pool exhausted: DBA team can increase temporarily. Alert at 80% pool utilisation should have fired.
    4. Check Oracle DB: sqlplus / as sysdba (from CBS host). Look for lock contention or tablespace full.
    5. Check CBS_BATCH_WINDOW_START — if during batch window (23:00-01:00), CBS may be under high load.
    Do not restart CBS during batch window without CBS Team approval.
    6. Review logs: /var/log/cbs/app.log
    ESCALATE: cbs-ops@company.com immediately. SEV-1 — CBS down affects all payment processing.

    EOD-001 → ETL-001 → RPT-001
    ↓ (quarterly)

  [FAILURE ESCALATION]
    EOD-001 failure → cbs-ops@company.com + CBS Manager (immediate)
    ETL-001 failure → Data Engineering on-call (PagerDuty: DE-ONCALL)
    RPT-001 failure → bi-ops@company.com
    Any job >2hr overrun → Platform Engineering auto-notified
    Airflow Dashboard: http://10.10.50.30:8081 (MGMT-NET only)
    Logs retained: 30 days at /var/log/airflow/

  [NIGHTLY JOB SEQUENCE]
    JOB: EOD-001 — End of Day Reconciliation
    Schedule: 23:00 daily
    System: CBS
    Duration: ~30 minutes
    Purpose: Closes business day. Calculates interest accruals, updates balances, generates EOD snapshot.
    Output: /data/exports/eod/YYYYMMDD_eod.csv
    On Failure: Alert to cbs-ops@company.com. CBS locked for transactions until resolved.
    Dependency: Must complete before ETL-001.
    JOB: ETL-001 — CBS to Data Warehouse Load
    Schedule: 23:30 daily
    System: Airflow DAG dag_cbs_to_dwh
    Duration: ~60 minutes
    Purpose: Extract CBS EOD snapshot, transform, load into Snowflake DWH.
    Input: /data/exports/eod/YYYYMMDD_eod.csv
    Output: DWH tables: fact_transactions, dim_accounts, dim_customers
    On Failure: Auto-retry once. If second failure, page Data Engineering on-call.
    Dependency: EOD-001 must complete.
    JOB: RPT-001 — Regulatory Reporting
    Schedule: 01:00 daily
    System: REPORTING-SVC
    Duration: ~45 minutes
    Purpose: Generate Basel III liquidity ratios, AML summary, transaction volume reports.

  [OVERVIEW]
    All scheduled batch jobs run during nightly processing window: 23:00-05:00.
    Orchestrated via Apache Airflow. Dashboard: http://10.10.50.30:8081 (MGMT-NET only).

  [KNOWN ETL ISSUES]
    KNOWN ETL ISSUES
    ETL-001 job failure:
    most common cause: CBS transaction export schema change without ETL team notification
    INC-2025-0288 root cause: CBS v4.2.0 added beneficiary_account_type column.
    ETL job failed because the Snowflake staging table didn't have this column.
    fix: DBA added the column + index. ETL resumed normally.
    if ETL-001 fails: check /var/log/etl/etl_run.log for schema mismatch errors
    second most common cause: Snowflake warehouse suspended (cost savings auto-suspend)
    DWH_SNOWFLAKE_WAREHOUSE = COMPUTE_WH - auto-suspends after 10 min idle
    ETL job resumes it on next run but first run after suspension takes 60-90 seconds longer
    this is normal, not an incident

  [ETL PIPELINE OVERVIEW]
    ETL PIPELINE OVERVIEW
    the main ETL job is: CBS → Kafka → DWH (Snowflake)
    - CBS exports transaction data via Kafka topic: cbs.transaction.export
    - ETL consumer (Python, runs on 10.10.50.60) reads from cbs.transaction.export
    - transforms and loads into Snowflake fact_transactions table
    - ETL job runs every 15 minutes via cron (/opt/scripts/etl/run_etl.sh)
    - ETL job name: ETL-001 (as referenced in operations runbook)

    Q: why is my report showing yesterday's data and not today's?
    A: the DWH (snowflake) is updated nightly by the ETL pipeline which runs from 23:30.
    so anything you run before that has data from the previous day (T+1 lag).
    if you need same-day data you need to query CBS directly - talk to CBS team for read access.
    if its past midnight and you still see old data, the ETL might have failed - check airflow
    at http://10.10.50.30:8081 or ask data eng.

  [---]
    ---
    Q: the auth service was down for 47 minutes in may 2024. what happened?
    A: INC-2024-0341. a junior engineer changed the AUTH-SVC config file manually (not via the
    CI/CD pipeline). the PostgreSQL password had a trailing whitespace. keycloak couldnt connect
    to its database. since ALL apps validate tokens against AUTH-SVC, 100% of user sessions
    dropped. 8400 active sessions lost.
    fix: all config changes now go through CI/CD pipeline only. manual config edits are blocked.
    keycloak health check now probes DB connectivity not just process health.

  [---]
    ---
    Q: who approves access to production database directly?
    A: production DB direct access requires sysadmin role which needs CISO approval + manager.
    time-limited to 90 days, all sessions recorded via PAM. not something we give out easily.
    in 99% of cases you should be using the application APIs or the DWH for data access.
    what specifically do you need? there might be an API that covers your use case.
