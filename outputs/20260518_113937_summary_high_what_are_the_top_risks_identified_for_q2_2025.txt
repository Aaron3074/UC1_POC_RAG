  [ISSUE: Reporting Service slow — timeout on report generation]
    STEPS:
    1. Check DWH query performance — review query plan in Snowflake console.
    2. Verify ETL-001 completed — stale DWH data causes report retries.
    3. Check resource usage: ssh 10.10.1.40, run top or htop.
    4. Check if CBS_BATCH_WINDOW_START overlaps with RPT-001 start time (I/O contention).
    ESCALATE: bi-ops@company.com

  [ISSUE: Fraud Engine high false positives blocking legitimate payments]
    STEPS:
    1. Verify PAYMENTS-GW_FRAUD_THRESHOLD. Default 0.75. Lower value blocks more payments.
    2. Check FRAUD_MODEL_VERSION — if recently updated, model may need recalibration.
    3. Check FRAUD_REDIS_TTL_SEC — stale cached scores can cause issues.
    4. Flush Redis cache if stale scores suspected (coordinate with Risk Tech team first):
    redis-cli -h 10.10.1.30 -p 6379 FLUSHDB
    5. Review logs: /var/log/fraud-engine/scoring.log
    ESCALATE: risktech@company.com

  [ISSUE: Payments Gateway not processing — HTTP 503/504]
    STEPS:
    1. Check health: GET http://10.10.1.20:9000/payments/health (expected: status ok)
    2. SSH to 10.10.1.20 from MGMT-NET. Check: systemctl status payments-gw
    3. Check FRAUD-ENGINE: curl http://10.10.1.30:8500/fraud/score/health
    PAYMENTS-GW calls FRAUD-ENGINE synchronously. If Fraud Engine is down, payments are blocked.
    4. Check CBS: curl http://10.10.1.10:8080/api/v2/health
    5. Review logs: /var/log/payments-gw/app.log — look for ConnectionRefused or Timeout errors.
    ESCALATE: If unresolved 15 minutes, page PAYMENTS-ONCALL (PagerDuty).

  [ISSUE: ETL-001 job failed — DWH showing stale data]
    STEPS:
    1. Check Airflow at http://10.10.50.30:8081 — find dag_cbs_to_dwh, review task logs.
    2. Common cause 1: EOD-001 did not complete. Check if /data/exports/eod/YYYYMMDD_eod.csv exists.
    3. Common cause 2: Snowflake DWH unreachable. Test connectivity from ETL host.
    4. Common cause 3: Schema mismatch — CBS added column not mapped. Check logs for KeyError.
    5. Manual re-trigger: In Airflow UI, clear and re-run dag_cbs_to_dwh for the failed date.
    ESCALATE: Data Engineering on-call if retry fails (PagerDuty: DE-ONCALL).

  [fraud model deployment notes - internal - risk tech team]
    fraud model deployment notes - internal - risk tech team
    compiled by: risk tech lead
    covers: fraud-engine model versioning, deployment gates, known constraints
    last updated: april 2025

  [MODEL PARAMETER CONSTRAINTS]
    MODEL PARAMETER CONSTRAINTS
    FRAUD_MODEL_VERSION = "3.2.1" (in env, updated by deploy script)
    FRAUD_SCORE_THRESHOLD = 0.75 (scores above this trigger fraud hold)
    - DO NOT change this without risk committee approval
    - last changed: Q3 2023 after false positive rate review
    FRAUD_KAFKA_CONSUMER_GROUP = "fraud-scoring-cg-prod" (do not rename, kafka offset tracking depends on this)
    if you rename the consumer group, offset tracking is lost and the fraud engine will reprocess
    every message in the partition from the beginning. this caused INC-2024-0991 in september 2024.
    FRAUD_REDIS_TTL_SEC = 3600
    - redis cache TTL for fraud score results. set to 3600 by platform eng.
    - risk tech has requested reducing to 1800 for Q2 to improve freshness of repeat transaction scoring
    - platform eng said NO until redis auto-scaling is confirmed (sentinel config done, load test pending)
    - do NOT change this even if risk tech asks. they know the constraint.
    FRAUD_BATCH_SIZE = 500 (kafka consumer batch size)
    FRAUD_MAX_RETRIES = 3 (retry on scoring failure before dead-letter queue)

  [KNOWN ISSUES AND GOTCHAS]
    KNOWN ISSUES AND GOTCHAS
    - FRAUD_REDIS_TTL_SEC sensitivity: lower TTL = more cache misses = higher latency on payments GW
    at 900 TTL, payments GW latency spiked from 120ms to 380ms in load test (jan 2025)
    this is why platform eng wont reduce TTL until auto-scaling is proven
    - kafka consumer group rename caused INC-2024-0991 (offset reset, reprocessing backlog)
    rule: FRAUD_KAFKA_CONSUMER_GROUP is immutable in prod
    - model warm-up: fraud engine needs 3-5 minutes after deploy before scoring performance stabilises
    during warm-up, FRAUD_SCORE_THRESHOLD is temporarily raised to 0.85 to reduce false positives
    this is handled automatically by the deploy script. dont panic if you see fewer holds in first 5 min.
    - v3.4.0 specifically: the new model uses an updated feature vector that includes beneficiary_account_type
    this field comes from CBS via the payments gateway. CBS v4.2.0+ is required for this field to be present.
    CBS is already on v4.2.1 in prod so this dependency is met. just noting it.
