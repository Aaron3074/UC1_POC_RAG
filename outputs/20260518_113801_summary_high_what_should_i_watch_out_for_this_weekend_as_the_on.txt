[4] Operations Troubleshooting Runbook  [.TXT]
  Topic  : Troubleshooting, Incident Response, Common Errors
  Owner  : Platform Engineering Team  |  v6.2  |  Updated: 2025-05-17

  [ISSUE: ETL-001 job failed — DWH showing stale data]
    STEPS:
    1. Check Airflow at http://10.10.50.30:8081 — find dag_cbs_to_dwh, review task logs.
    2. Common cause 1: EOD-001 did not complete. Check if /data/exports/eod/YYYYMMDD_eod.csv exists.
    3. Common cause 2: Snowflake DWH unreachable. Test connectivity from ETL host.
    4. Common cause 3: Schema mismatch — CBS added column not mapped. Check logs for KeyError.
    5. Manual re-trigger: In Airflow UI, clear and re-run dag_cbs_to_dwh for the failed date.
    ESCALATE: Data Engineering on-call if retry fails (PagerDuty: DE-ONCALL).

  [GENERAL CONTACTS]
    Platform Engineering: platform-eng@company.com | PagerDuty: PE-ONCALL
    Network Security: security-ops@company.com
    CBS Team: cbs-ops@company.com
    Data Engineering: data-eng@company.com | PagerDuty: DE-ONCALL
    Payments Team: PagerDuty: PAYMENTS-ONCALL
    Risk Technology: risktech@company.com
    BI Team: bi-ops@company.com

  [ISSUE: Payments Gateway not processing — HTTP 503/504]
    STEPS:
    1. Check health: GET http://10.10.1.20:9000/payments/health (expected: status ok)
    2. SSH to 10.10.1.20 from MGMT-NET. Check: systemctl status payments-gw
    3. Check FRAUD-ENGINE: curl http://10.10.1.30:8500/fraud/score/health
    PAYMENTS-GW calls FRAUD-ENGINE synchronously. If Fraud Engine is down, payments are blocked.
    4. Check CBS: curl http://10.10.1.10:8080/api/v2/health
    5. Review logs: /var/log/payments-gw/app.log — look for ConnectionRefused or Timeout errors.
    ESCALATE: If unresolved 15 minutes, page PAYMENTS-ONCALL (PagerDuty).

  [ISSUE: Fraud Engine high false positives blocking legitimate payments]
    STEPS:
    1. Verify PAYMENTS-GW_FRAUD_THRESHOLD. Default 0.75. Lower value blocks more payments.
    2. Check FRAUD_MODEL_VERSION — if recently updated, model may need recalibration.
    3. Check FRAUD_REDIS_TTL_SEC — stale cached scores can cause issues.
    4. Flush Redis cache if stale scores suspected (coordinate with Risk Tech team first):
    redis-cli -h 10.10.1.30 -p 6379 FLUSHDB
    5. Review logs: /var/log/fraud-engine/scoring.log
    ESCALATE: risktech@company.com
